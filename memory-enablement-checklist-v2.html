Four-Tier Retrieval Model
Every piece of state the agent can access sits in one of four tiers. Each tier has a distinct volatility, authorship, integrity profile, and retrieval channel.
T1Short-Term
Memory
Platform auto-extract between sessions. Volatile, lossy, model-authored. No checksum, no version. Context hints only — never authoritative.
Channel: injected by platform · Scope: cross-session, project-limited · Examples: inferred prefs, session summaries, observed behaviors
▼ authority + integrity increase · volatility decreases ▼
T2Chat
History
Verbatim transcript index, searchable via conversation_search and recent_chats. Not model-authored — records exactly what was said. Indefinite retention within project scope. Evidence-grade, not directive-grade.
Channel: agent-invoked search tools · Scope: project-limited, indefinite · Examples: past commands, file MD5s, decisions, quoted errors
▼▼
T3Session
Files
Human-authored files: project knowledge, uploads, skills, checkpoints. Versioned in-context, checksummable (MD5), subject to re-read warning protocol. Authoritative in-session. See claude.md §1–3.
Channel: project injection + /mnt/user-data/uploads/ · Scope: session to project · Examples: claude.md, checkpoints, tachikoma-framing-draft
▼▼▼
T4Git
Repositories
External version-controlled source of truth. Diffable, blame-able, commit-SHA-addressable. Survives project deletion, platform migration, and memory resets. Canonical across time. Requires trusted-host entry.
Channel: git clone/fetch/push via github.com trusted host · Scope: org/repo-level, durable · Examples: ecological-ai/user-prefs, axiomatic-cmd/ai-therapy

Cross-Tier Conflict Resolution

When two tiers disagree on state (e.g., a memory claims v1.3.3 but a repo tag shows v1.3.5), the higher tier wins. Agent surfaces the conflict before resolving silently.

Conflict Winner Action
T4 vs T3T4 (repo)Repo commit SHA + tag is canonical. Update file locally.
T3 vs T2T3 (file)File in context overrides stale chat transcript claims.
T3 vs T1T3 (file)Per claude.md §7.2. Flag memory for deletion.
T2 vs T1T2 (history)Transcript is evidence; memory is inference.
T4 vs T1T4 (repo)Repo wins decisively. Memory is stalest of all tiers.

Ongoing Hygiene — Enabled State
Recurring maintenance across all four tiers.

Monthly memory audit (T1)

Review stored memories every 4–6 weeks. Delete stale, redundant, or sensitive entries. Treat as cache — prune aggressively. Next audit due ~May 13, 2026.

human

Four-tier conflict surfacing at session start

Per claude.md §7.3: agent flags contradictions across T1/T2/T3/T4 before proceeding. Silent resolution prohibited even when precedence is obvious.

rule agent

No secrets in any tier

Per claude.md §7.5 + §7.5.1: API keys, PATs, passwords, tokens, hostnames, IPs — never in T1 (memory), T3 (project files), or T4 (repos). Credential material enters via upload-only, runtime env var only. See §7.5.1.1 file-upload + bash-pipe pattern.

rule

Transcript secret scan (T2 hygiene)

conversation_search indexes user messages verbatim. If a secret was ever pasted inline, it persists in T2 indefinitely within project scope. Periodically search for patterns (github_pat_, sk-, Bearer, API key prefixes). Any hit → rotate credential + request transcript deletion via platform support.

new v2.0 human

Cross-project bleed review (T1 + T2)

Check if memories or chat history from other projects influence current session. Memory scope is project-limited by design; verify empirically. Chat history search is already project-scoped — no cross-project reach.

human agent

Chat history scope discipline (T2)

Agent must never assume transcript content is authoritative. T2 is evidence of what was said — not evidence of what is true. When T2 conflicts with T3/T4, defer to higher tier. When T2 is the only channel available, cite verbatim + flag as unverified.

new v2.0 rule

Public-repo content gating (T4)

Before pushing to a public repo: verify no secrets, no internal hostnames, no PII, no project-private context. Run secret-scan pre-commit (git-secrets, trufflehog, or grep against known prefix list).

new v2.0 rule
4
Positional Trust & Canonization
How files earn elevated authority over time.

Track file stability in checkpoint metadata

Per claude.md §7.4: record file ages, session-load counts, and stability periods. Files surviving many sessions unmodified earn higher implicit trust.

agent

T3 canonical threshold

A T3 file becomes "canonical" when: loaded in 10+ sessions, checksum stable for 30+ days, and human-reviewed at least once during that period.

rule

T3 canonical → agent-side trust elevation

Canonical T3 files get maximum positional trust against lower tiers (T1, T2). Conflict with a memory or fresh file → canonical wins. Agent flags but does not override without explicit instruction.

rule

T4 supersedes T3 on commit

Once a T3 file is pushed to a trusted-host repo with a signed or tagged commit, T4 becomes canonical for that file. Local T3 copies are mirrors — if drift is detected, T4 wins and T3 is updated from repo.

new v2.0 rule
5
Git Promotion Protocol New v2.0
Promoting a T3 file to T4 canonical status.

Promotion trigger criteria

Promote T3 → T4 when: (a) file has reached T3 canonical threshold, (b) content is stable enough to warrant external archival, and (c) public/private visibility has been decided (see content gating in Phase 3).

rule

Trusted-host entry required

Before first push to a new repo host, add an entry in trusted-hosts.md with appropriate trust_level, allowed_methods, and verified status. Current entries: github.com (git HTTPS transport), api.github.com (REST; note proxy gap).

rule human

PAT two-channel verification (claude.md §7.5.1.1)

Channel 1 (agent, in-session): MD5 of uploaded PAT file + REST scope check. Channel 2 (human, out-of-band): if Channel 1 blocked by harness proxy, human confirms scope, expiry, permissions. If both channels fail → default to mandatory post-session rotation. Log channel used in checkpoint.

new v2.0 agent

Commit SHA referenced in checkpoints

When a T4-canonical file is updated, the owning checkpoint records the commit SHA (not just MD5). Future sessions can retrieve the exact version via git checkout <sha>. MD5 attests content; SHA attests provenance + position in history.

new v2.0 agent

Scrub PAT from .git/config post-push

After git push, rewrite remote URL to strip embedded PAT: git remote set-url origin https://github.com/<owner>/<repo>.git. Prevents PAT lingering in workspace config for remainder of session.

rule agent

Rotation posture per claude.md §7.5.1.1

File-upload + bash-pipe pattern + fine-grained PAT (repo-scoped, minimum perms, ≤90d expiry, no refresh token) → delegated rotation to encoded controls. Inline-paste or overscoped token → mandatory post-session rotation. Channel 2 unavailable → default mandatory.

rule

Known Limitation — Prompt Injection Across All Tiers

Every tier delivers tokens into the context window. Adversarial content in a fetched URL, uploaded file, MCP tool output, or even a commit message can mimic a [RULE] directive or memory. No cryptographic defense exists within the current platform.

Mitigations above — positional trust (T4 > T3 > T2 > T1), file-wins-over-memory precedence, two-channel verification, trusted-host allowlist — reduce risk for common cases but do not eliminate adversarial injection. Higher tier does not equal higher safety; a compromised repo is a canonical source of compromise.

Known Limitation — T2 Chat History Persistence

Transcripts are indexed verbatim within project scope, indefinitely. Any secret ever pasted inline remains retrievable via conversation_search from any session in that project, regardless of whether the originating chat is visible in the sidebar. Platform-side deletion of individual messages is not currently exposed to users in a granular form. Prefer file-upload + bash-pipe over inline paste, always.

Known Limitation — T4 Depends on External Service

Git repo availability depends on the hosting provider (github.com). Outages, account suspensions, or org transfers can make canonical state temporarily unretrievable. Pin critical files locally (T3 mirrors of T4) for resilience. Consider mirroring to a second host for tier-4-critical artifacts.
