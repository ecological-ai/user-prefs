Four-Tier Memory Classification
Per claude.md §7.1. Each tier has distinct authorship, integrity profile, and retrieval channel. Tiers are not strictly ordered by authority — precedence depends on content source and verification (see conflict table below).
Short-TermMemory
Platform auto-extracted between sessions. Volatile, lossy, model-authored. No checksum, no version. Context hints only — never authoritative.
Channel: injected by platform · Scope: cross-session, project-limited · Examples: inferred prefs, session summaries, observed behaviors
Long-TermMemory
Human-authored files loaded explicitly. Versioned, checksummed, integrity-validated. Authority grows with temporal stability + human review. Most stable form: git-version-controlled commits.
Channel: project injection + /mnt/user-data/uploads/ + git clone/fetch via trusted hosts · Scope: session to durable · Examples: claude.md, prompteng-SKILL.md, checkpoints, framing drafts
Sub-tiers by substrate & durability
  • Session files — project knowledge, uploads, skills loaded into current context. In-session authoritative. MD5-registered per §1–3.
  • Git repositories — external version-controlled source of truth. Diffable, blame-able, commit-SHA-addressable. Survives project deletion, platform migration, memory resets. Canonical across time.
SelectiveMemory
On-demand retrieval of data from chat history (if enabled in platform settings). User-driven retrieval used for establishing better context in a conversation. Verbatim transcript index — records exactly what was said. Indefinite retention within project scope. Evidence-grade, not directive-grade.
Channel: agent-invoked conversation_search + recent_chats · Scope: project-limited, indefinite · Examples: past commands, file MD5s, decisions, quoted errors
LatentMemory
Facts embedded in the training data of the language model in use. Authoritative nature of this tier must be evaluated at runtime when required — via web-search grounding, file-based sources, or explicit human confirmation. Cannot be checksummed, versioned, or scoped.
Channel: model inference (implicit in every token) · Scope: model-global, pre-training cutoff · Examples: general world knowledge, library APIs, code patterns, historical facts

Cross-Tier Conflict Resolution

Per claude.md §7.2. When tiers disagree, verifiable substrate wins. Agent surfaces conflict before resolving silently.

Conflict Winner Action
Git repo vs Session file (within Long-Term) Git repo Repo commit SHA + tag is canonical. Update session file locally to match.
Long-Term vs Short-Term Long-Term Per claude.md §7.2. Flag memory for deletion if stale.
Long-Term vs Selective Long-Term File is directive; history is evidence. File wins unless file explicitly defers to transcript.
Selective vs Short-Term Selective Transcript is evidence-grade; memory is inference. Cite verbatim.
Long-Term vs Latent Long-Term Explicit human-authored content overrides training priors. Latent knowledge is default, not directive.

Cross-tier conflict surfacing at session start

rule agent

No secrets in any tier

Per claude.md §7.5 + §7.5.1: API keys, PATs, passwords, tokens, hostnames, IPs — never in Short-Term (memory), Long-Term (project files, git repos). Credential material enters via upload-only, runtime env var only. See §7.5.1.1 file-upload + bash-pipe pattern.

rule

Selective (chat history) secret scan

conversation_search indexes user messages verbatim. If a secret was ever pasted inline, it persists in the Selective tier indefinitely within project scope. Periodically search for patterns (github_pat_, sk-, Bearer, API key prefixes). Any hit → rotate credential + request transcript deletion via platform support.

new v3.0 human

Cross-project bleed review (Short-Term + Selective)

Check if memories or chat history from other projects influence current session. Especially high risk when loading checkpoint files from several projects. Memory scope is project-limited by design; verify empirically. Chat history search is project-scoped — no cross-project reach by default.

human agent

Selective-tier scope discipline

Agent must never assume transcript content is authoritative. Selective tier is evidence of what was said — not evidence of what is true. When Selective conflicts with Long-Term, defer to Long-Term. When Selective is the only channel available, cite verbatim + flag as unverified.

rule

Latent-tier verification discipline

Claims grounded only in Latent memory (training data) must be flagged when they concern fast-changing facts (current events, recent API changes, evolving standards). Re-verify via web search, file-based sources, or explicit human confirmation before acting on Latent claims.

new v3.0 rule

Public-repo content gating (Long-Term git sub-tier)

Before pushing to a public repo: verify no secrets, no internal hostnames, no PII, no project-private context, no unintended IP disclosure. Run secret-scan pre-commit (git-secrets, trufflehog, or grep against known prefix list).

4
Positional Trust & Canonization
How Long-Term files earn elevated authority over time.

Track file stability in checkpoint metadata

Per claude.md §7.4: record file ages, session-load counts, and stability periods. Files surviving many sessions unmodified earn higher implicit trust.

agent

Canonical threshold (updated in v1.5.2)

A Long-Term file becomes "canonical" when: loaded in 12+ sessions, checksum stable for 60+ days, and human-reviewed at least once during that period. Raised from 10 sessions / 30 days in prior version.

updated v3.0 rule

Canonical → agent-side trust elevation

Canonical Long-Term files get maximum positional trust against lower tiers (Short-Term, Selective). Conflict with a memory or fresh file → canonical wins. Agent flags but does not override without explicit instruction.

rule

Git sub-tier supersedes session-file sub-tier on commit

Once a session file is pushed to a trusted-host repo with a tagged commit, the git sub-tier becomes canonical for that file. Local session-file copies are mirrors — if drift is detected, git wins and local is updated from repo.

rule
5
Git Promotion Protocol v2.0
Promoting a session file to git-backed canonical status.

Promotion trigger criteria

Promote session file → git sub-tier when: (a) file has reached canonical threshold, (b) content is stable enough to warrant external archival, and (c) public/private visibility has been decided (see content gating in Phase 3).

rule

Trusted-host entry required

Before first push to a new repo host, add an entry in trusted-hosts.md with appropriate trust_level, allowed_methods, and verified status. Current entries: github.com (git HTTPS transport), api.github.com (REST; note proxy gap).

rule human

PAT two-channel verification (claude.md §7.5.1.1)

Channel 1 (agent, in-session): MD5 of uploaded PAT file + REST scope check. Channel 2 (human, out-of-band): if Channel 1 blocked by harness proxy, human confirms scope, expiry, permissions. If both channels fail → default to mandatory post-session rotation. Log channel used in checkpoint.

agent

Fetch before push — no rebase without approval

AIT-10 Task A amendment: before every push, git fetch origin <branch>. Compare local HEAD to remote HEAD. Remote ahead → halt, surface divergence, await human choice (rebase, merge, abort). Force-push always requires explicit human approval.

new v3.0 rule agent

Commit SHA referenced in checkpoints

When a git-canonical file is updated, the owning checkpoint records the commit SHA (not just MD5). Future sessions can retrieve the exact version via git checkout <sha>. MD5 attests content; SHA attests provenance + position in history.

agent

Scrub PAT from .git/config post-push

After git push, rewrite remote URL to strip embedded PAT: git remote set-url origin https://github.com/<owner>/<repo>.git. Prevents PAT lingering in workspace config for remainder of session.

rule agent

Rotation posture per claude.md §7.5.1.1

File-upload + bash-pipe pattern + fine-grained PAT (repo-scoped, minimum perms, ≤90d expiry, no refresh token) → delegated rotation to encoded controls. Inline-paste or overscoped token → mandatory post-session rotation. Channel 2 unavailable → default mandatory.

rule

Known Limitation — Prompt Injection Across All Tiers

Every tier delivers tokens into the context window. Adversarial content in a fetched URL, uploaded file, MCP tool output, or commit message can mimic a [RULES] directive or memory. No cryptographic defense exists within the current platform.

Mitigations above — positional trust, file-wins-over-memory precedence, two-channel verification, trusted-host allowlist — reduce risk for common cases but do not eliminate adversarial injection. Higher tier does not equal higher safety; a compromised repo is a canonical source of compromise.

Known Limitation — Selective-Tier Persistence

Transcripts are indexed verbatim within project scope, indefinitely. Any secret ever pasted inline remains retrievable via conversation_search from any session in that project, regardless of whether the originating chat is visible in the sidebar. Platform-side deletion of individual messages is not currently exposed to users in a granular form. Prefer file-upload + bash-pipe over inline paste, always.

Known Limitation — Git Sub-Tier Depends on External Service

Git repo availability depends on the hosting provider (github.com). Outages, account suspensions, or org transfers can make canonical state temporarily unretrievable. Pin critical files locally (session-file mirrors of git sub-tier) for resilience. Consider mirroring to a second host for critical artifacts.

Known Limitation — Latent Tier Staleness

Latent memory reflects training data up to a fixed cutoff. Claims about current events, recent library versions, changed APIs, deprecated endpoints, or evolving policy cannot be served from Latent alone. Ground every time-sensitive claim in a verifiable Long-Term or Selective source, or flag uncertainty explicitly. Confident-sounding Latent recall is the dominant confabulation mode.
